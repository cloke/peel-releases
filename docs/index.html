What Peel helps you do

The repo stays at the center. Everything else hangs off the code it's meant to improve.

01

Understand a repository

Search your code by meaning instead of filename. Indexing happens locally, so nothing leaves the machine.

02

Run agent workflows safely

Every run gets its own git worktree. You see the plan, the diff, and the checks before deciding what merges.

03

Keep one attention queue

The Inbox collects whatever needs you. Finished runs, failures, and PRs waiting on review all land in one place.

04

Automate recurring work

Save a workflow as a template, then put it on a schedule. Patrols keep working in the background.

05

Use hardware you already own

Got a second Mac sitting idle? Peel routes work to it, including local model inference.

06

Connect the agents you use

Peel speaks MCP and the OpenAI API, so your existing tools can talk to it without changing how you work.

Point your agents at Peel

Peel runs a local server that speaks two protocols. Which one you want depends on whether you're after the tools or the models.

Tools, over MCP

Point Claude Code or Cursor at the local endpoint. Peel's repository and git tooling shows up inside the client you already use.

Models, over the OpenAI API

Peel also serves models on an OpenAI-compatible endpoint. One base URL gets any OpenAI client whatever we're running locally, and Peel picks the model if you'd rather not.

Setup for each client lives in the repo. We're building a proper Integrations screen for this.

What is a swarm?

A swarm is a few Macs running Peel together. Each one advertises what it has: repos, hardware, models. Peel sends a task to whichever machine is best equipped to run it.

One Mac works fine on its own. A swarm on your LAN doesn't need an account either. Read the swarm guide
